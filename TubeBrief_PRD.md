# 제품 요구사항 정의서 (PRD) — TubeBrief

> **버전** v3.0 (구현 완료본) · **MVP 완성일** 2026-06-07 · **전략** 백엔드 최소화 + 자동화 파이프라인 + LLM 출력 스키마 강제
>
> **GitHub**: https://github.com/leeyongjoo9-rgb/tubebrief_mvp · **Service**: https://tubebrief-mvp.vercel.app/

---

## 1. 제품 개요

- **서비스명(가칭):** TubeBrief — 유튜브 신규 콘텐츠 자동 요약 및 아카이빙 서비스
- **한 줄 정의:** 사용자가 등록한 유튜브 채널의 신규 영상이 올라오면, 스케줄러가 **자동으로** 감지하여 AI가 **주제·출연자·타임스탬프별 구간까지 구조화한 깊이 있는 요약**을 생성하고 텍스트로 아카이빙해 대시보드에 보여주는 웹 서비스.
- **목표:** 2주 내에 "등록 → 자동 감지 → 요약 → 아카이빙 → 조회"의 전체 파이프라인이 사람 개입 없이 한 바퀴 도는 MVP를 완성한다.
- **핵심 가치:** "영상을 보지 않고도, 새 콘텐츠가 올라오면 알아서 **구조화된 요약본**(주제·출연자·타임스탬프)이 쌓인다."

---

## 2. 핵심 타겟 및 가치

- **타겟:** 유튜브에서 구독·관심 채널의 새 콘텐츠를 챙겨보고 싶지만 시간이 부족한 **모든 사람**. (특정 분야 종사자에 한정하지 않음 — 뉴스·교육·취미·업무 정보 등 폭넓게 적용.)
- **해결하는 문제:** 구독 채널이 많아질수록 "새 영상이 올라온 것조차" 놓친다. 봐야 할 영상은 쌓이는데 시간은 없다.
- **차별점:** 단순 요약 도구가 아니라 **(1) 자동 감지** + **(2) 깊이 있는 구조화 요약**이 핵심.
  - *자동 감지:* 사용자가 새로고침하지 않아도 백그라운드에서 신규 영상이 쌓이고 요약된다.
  - *깊이 있는 요약:* 핵심 내용뿐 아니라 **주제(토픽), 출연자 정보, 타임스탬프별 구간 요약**까지 구조화해 제공한다. "1분 요약"을 넘어 영상의 지도를 보여준다.

## 3. 핵심 기능 요구사항

> v1 대비 주요 변경: **(a) 자동 감지 트리거를 P0로 승격**, **(b) 자막 추출 실패 폴백 설계 명문화**, **(c) 키워드 기능을 P2로 강등**.

| 우선순위 | 기능명 | 상세 내용 | 구현 메모 |
|---|---|---|---|
| **P0** | 자동 감지 스케줄러 | 등록된 채널을 **정기적으로(예: 1~6시간 주기)** 폴링하여 신규 영상을 사람 개입 없이 감지·처리. | **이 서비스의 핵심.** Vercel Cron Jobs 또는 Supabase pg_cron + Edge Function. 무료 플랜의 실행 주기 제한 사전 확인 필요. |
| **P0** | 유튜브 신규 감지 (RSS) | 채널 ID 기반 RSS 피드(`youtube.com/feeds/videos.xml?channel_id=...`)를 파싱해 최신 영상 목록 획득. `published` 시각으로 신규 여부 판별. | API 인증·쿼터 불필요. 단, **RSS는 채널의 최근 15개만** 노출되므로 폴링 주기를 너무 길게 잡으면 누락 위험. |
| **P0** | 중복 방지 | 이미 처리한 영상은 다시 요약하지 않음. | DB에 `video_id` **UNIQUE 제약**. 감지 시 존재 여부 먼저 확인. |
| **P0** | AI 요약 & 저장 (폴백 포함) | 자막을 추출해 LLM으로 요약 후 저장. | 아래 **4. 요약 파이프라인 & 폴백 전략** 참조. 요약 호출은 **반드시 서버 사이드**에서만. |
| **P0** | 대시보드 UI | 요약 카드 그리드, 채널/날짜별 필터, 카드 클릭 시 상세 요약 + 원본 링크. | v0.dev로 초안 생성 → Cursor로 이식. |
| **P1** | 채널 등록/관리 | 사용자가 채널 URL을 추가/삭제. URL → channel_id 변환 처리. | 핸들(@name)·커스텀 URL은 channel_id로 변환하는 단계 필요. |
| **P1** | 자동 감지 상태 표시 | "마지막 확인: N분 전", 처리 실패한 영상 표시. | 자동화가 실제로 도는지 사용자/평가자가 신뢰할 수 있게. |
| **P2** | 키워드 기반 감지 | 키워드로 신규 영상 탐색. | **RSS로 구현 불가.** YouTube Data API(`search.list`)가 필요하며 쿼터·인증이 붙음. 2주 내라면 **드롭 또는 마지막에.** |
| **P2** | 텍스트 내보내기 | 요약본을 `.txt`/`.md`로 다운로드. | 프론트 JS 몇 줄. |

## 4. 요약 파이프라인 & 폴백 전략 (리스크 핵심)

### 4-1. 요약 출력 구조 (깊이 있는 요약)

단순 한 단락 요약이 아니라, 차별점에 맞춰 다음 항목을 **구조화된 형태(JSON 등)**로 생성·저장한다. LLM 프롬프트에서 이 스키마를 명시하면 안정적으로 뽑힌다.

- **핵심 요약:** 3~5줄 한눈 요약.
- **주제(토픽):** 영상이 다루는 주제·키워드 태그.
- **출연자 정보:** 등장 인물/진행자/게스트와 (자막·설명문에서 식별 가능한 경우) 간단한 소개.
- **타임스탬프별 구간 요약:** 구간(시작 시각) + 해당 구간 요약. 원본 영상의 해당 시점으로 점프하는 링크(`youtube.com/watch?v=...&t=123s`)와 연결.

> *주의:* 출연자 정보와 정확한 타임스탬프는 자막 품질에 의존한다. 자동생성 자막에는 화자 구분이 없으므로, 출연자 식별은 "자막+설명문에서 추정"으로 처리하고 불확실하면 비워둔다(환각 방지). 타임스탬프는 자막의 시간 정보를 활용해 구간을 나눈다.

### 4-2. 폴백 전략

자막 추출은 **가장 깨지기 쉬운 구간**이다. 클라우드 IP에서 유튜브가 봇으로 차단하거나, 자막이 아예 없는 영상도 흔하다. 따라서 단일 경로에 의존하지 않고 단계적 폴백을 설계한다.

1. **1순위 — 자막 추출:** `youtube-transcript` 등으로 자막(수동 우선, 없으면 자동생성) 추출 → LLM 요약. 긴 영상(1시간+)은 **청크 분할 후 맵-리듀스 요약**으로 토큰 한도 대응.
2. **2순위 — 메타데이터 요약 (폴백):** 자막 추출 실패 시 제목 + 설명문만으로 **핵심 요약·주제만** 생성하고(타임스탬프·출연자 구간 요약은 생략), 카드에 **"자막 없음 — 제한적 요약"** 배지를 표시. 품질이 낮음을 사용자에게 투명하게 알린다.
3. **3순위 — 실패 기록:** 둘 다 실패하면 `status=failed`로 저장하고 원본 링크만 노출. 파이프라인이 멈추지 않게 한다.

> **데모 안정성 팁:** 발표/시연 전, 자막이 확실히 있는 채널 몇 개를 미리 등록해 "성공 경로"를 보장하되, 폴백이 동작하는 사례도 1개 준비해 견고함을 보여준다.

## 5. 사용자 여정 (User Flow)

1. 웹사이트 접속 → 대시보드에서 **자동으로 쌓인** 최근 요약 카드 목록 확인.
2. 카드 클릭 → 상세 요약본(핵심 요약 + 주제 + 출연자 + 타임스탬프별 구간) + 원본 영상 링크 + "자막 기반/메타데이터 기반" 표시. 타임스탬프 클릭 시 해당 시점으로 영상 점프.
3. 설정 페이지에서 구독할 채널 URL 등록/삭제.
4. 이후 사용자가 아무것도 하지 않아도, **스케줄러가 주기적으로 신규 영상을 감지·요약해 대시보드에 추가**.

## 6. 데이터 파이프라인 (자동화 흐름)

```
[Cron 스케줄러] ──주기 실행──> [채널별 RSS 파싱]
        │
        ▼
[신규 video_id 판별] ──이미 있음──> (건너뜀)
        │ 신규
        ▼
[자막 추출] ──성공──> [LLM 요약(청크/맵리듀스)]
        │ 실패              │
        ▼                   ▼
[메타데이터 폴백 요약]   [Supabase 저장(UNIQUE: video_id)]
        │                   │
        └────────┬──────────┘
                 ▼
        [대시보드 조회 / 필터 / 내보내기]
```

## 8. 보안 & 비용 주의사항

- **API 키 노출 금지:** LLM·외부 API 키는 `.env.local` 환경변수로만. 프론트 코드(v0.dev 결과물 포함)에 절대 하드코딩하지 않는다. 모든 요약 호출은 API Route(서버)에서 처리.
- **토큰 비용 관리:** 긴 영상은 청크 분할, 저비용 모델(Haiku/mini) 기본 사용.
- **중복 호출 방지:** UNIQUE 제약 + 처리 전 존재 확인으로 같은 영상의 재요약(=비용 낭비) 차단.

## 9. 2주 완성 스케줄 — 진행 결과

**1주차 — 인프라 + 파이프라인 (계획대로 진행)**
- Next.js + Tailwind + shadcn/ui + Supabase 초기화
- DB 스키마: `channels`, `videos` (+ 사용자 추가 `subscriptions`, `transcripts`) 생성
- RSS 신규 감지 + 자막 추출 + 메타데이터 폴백 + LLM 구조화 요약 → 전체 파이프라인 작동
- 영상 88+ 건 실제 처리 검증

**2주차 — 자동화 + 품질 개선 + 배포**
- `/api/cron` 통합 엔드포인트 — refresh → process → summarize 직렬 실행
- Vercel Cron 일 1회 자동 트리거 등록 (Hobby 제약 60초 안에 들어가도록 limit 보수적 설정)
- 채널 관리 UI (`/channels`) 추가 — 폼으로 채널·구독 등록·삭제, 서버 액션으로 비밀 키 미노출
- **요약 품질 반복 개선**: 옛 규칙(brief 290자 평균) → 강화된 규칙(brief 2000자대) → 사용자 검증 후 `mentioned_people`·`companies` 필드 추가
- GitHub push → Vercel 배포 → 첫 cron 검증 완료

**완성 기준 충족 여부**
| 항목 | 기준 | 결과 |
|---|---|---|
| 자동 감지 | 등록한 채널의 신규 영상이 사람 개입 없이 잡힘 | ✅ Vercel Cron 일 1회 실측 검증 |

| 환각 방지 | 자막에 없는 정보 만들지 않음 | ✅ 인물 식별률 검증 — 정보 없는 영상은 빈 배열로 둠 |

| 자동화 상태 표시 | "마지막 확인" 시각 노출 | ✅ 홈 헤더에 `subscriptions.last_checked` MAX 표시 |

## 10. v1 대비 변경 요약

- **추가/승격:** 자동 감지 스케줄러(P0), 중복 방지 UNIQUE 제약(P0), 자막 폴백 3단계 전략, 자동화 상태 표시(P1), 데이터 파이프라인 다이어그램, **깊이 있는 구조화 요약(주제·출연자·타임스탬프) 출력 스키마**.
