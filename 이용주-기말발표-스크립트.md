## 슬라이드 2 — 문제정의와 대상 사용자 (40초)

> 이 서비스의 출발점은 단순한 문제입니다. **구독 채널이 늘어날수록 무엇부터 봐야 할지 판단이 어려워진다** 는 점입니다.
>
> 저처럼 8개 이상 채널을 구독하면 매일 영상이 5편에서 많으면 20편씩 쌓입니다. 어떤 영상이 가치 있는지 미리 알 수 없고, 알고리즘 추천은 본인 관심사와 어긋날 때가 많습니다.
>
> 그런데 유튜브가 제공하는 자막은 길고 비구조화 텍스트라 훑어 읽기에 부적합하고, 흔한 1줄 요약 도구들은 정보가 너무 부족합니다.
>
> 저희가 잡은 가설은 — **"매일 자동으로 한 장짜리 구조화 보고서가 쌓이면, 1분 안에 영상의 핵심을 파악해서 골라볼 수 있다"** 는 겁니다.

## 슬라이드 3 — 서비스 개요와 핵심 기능 (40초)

> 한 문장으로 요약하면, **"매일 자동 감지 → 자막+LLM → 구조화 한 장 보고서 → 대시보드 아카이빙"** 입니다.
>
> 핵심 기능 3개입니다.
>
> **첫째**, 자동 감지 cron. Vercel Cron 이 매일 0시에 깨어나서 RSS → 자막 추출 → LLM 요약까지 직렬로 실행합니다. 사용자 개입은 0회입니다.
>
> **둘째**, 코너 단위 구독. 예를 들어 MBC 라디오 시사 채널처럼 한 채널에 7개 프로그램이 섞여 있을 때, 키워드 AND 필터로 *권순표의 뉴스하이킥* 만 받을 수 있습니다. 호스트 이름이 제목에 없고 설명문 해시태그에만 있는 경우도 잡아냅니다.
>
> **셋째**, 출연자와 거론된 인물 분리. LLM 이 직접 출연한 게스트와 대화 중 거론된 역사적 인물을 구분해서 별도 필드로 저장합니다.

## 슬라이드 4 — 사용자 이용 흐름 (55초)

> 사용자 이용 흐름을 보여드리겠습니다.
>
> **입력 단계** 에서 사용자는 채널 관리 페이지에 채널 URL 과 선택적으로 키워드 필터를 한 번 등록합니다. 그 뒤로 사람의 개입은 없습니다.
>
> **자동 처리 단계** 는 매일 0시 UTC 에 다섯 단계로 진행됩니다. Vercel Cron 이 깨어나서, RSS 폴링으로 신규 영상을 발견하고, youtube-transcript 로 자막을 뽑고, 실패하면 RSS 설명문 메타데이터로 폴백합니다. 그 뒤 gpt-5-mini 가 JSON 스키마에 맞춰 요약을 생성하고, Supabase 의 jsonb 컬럼에 저장합니다.
>
> **결과 단계** 에서 사용자는 홈 대시보드의 ChannelStrip 으로 채널을 필터링하고, 카드를 클릭해 상세 페이지로 들어갑니다. 상세 페이지에서는 본문 안에 자연스럽게 삽입된 시각 마커 — `(12:34)` 같은 — 를 클릭하면 그 시점의 유튜브로 바로 점프할 수 있습니다.

## 슬라이드 6 — 기술스택 및 구현 구조 (35초)

> 기술스택은 단일 Next.js 프로젝트 안에 다 통합돼 있습니다.
>
> **Framework** 는 Next.js 16 App Router — API Routes 로 백엔드까지 한 프로젝트에서 처리합니다.
>
> **DB** 는 Supabase 의 PostgreSQL. LLM 의 JSON 응답을 jsonb 컬럼에 그대로 저장할 수 있어서 별도 매핑이 필요 없습니다.
>
> **LLM** 은 OpenAI gpt-5-mini — 핵심은 **structured outputs strict mode** 입니다. JSON 스키마를 OpenAI 가 강제하기 때문에 환각이나 포맷 깨짐이 원천 차단됩니다.
>
> **배포** 는 Vercel Hobby. GitHub 에 push 하면 자동 빌드되고, Vercel Cron 으로 일 1회 자동 실행 — 모두 무료 tier 입니다.

## 슬라이드 7 — MCP/API 활용 방식 (40초)

> 사용한 API 는 4개 + 인프라 플랫폼 1개입니다.
>
> **OpenAI Chat API** 는 가장 중요합니다. 자막을 시스템 프롬프트, JSON Schema 와 함께 보내서 구조화된 응답을 받습니다. strict 모드라 스키마 위반이 불가능합니다.
>
> **Supabase** 는 server-side 에서는 service role key 로, 클라이언트에서는 publishable key 로 RLS 보호 아래 분리해 접근합니다.
>
> **YouTube RSS** 는 API 키 없이 무료로 신규 영상 감지에 쓰고, **YouTube 페이지 스크래핑** 은 URL 에서 channel_id 를 추출할 때 6가지 정규식 패턴으로 fallback 합니다.
>
> 그리고 **Vercel Platform** 자체를 인프라 API 로 활용합니다 — Cron 자동화, Serverless Functions, 환경변수 관리, GitHub 자동 배포까지.

## 슬라이드 8 — 컨텍스트 엔지니어링 활용 방식 (40초)

> Claude Code 를 협업 도구로 쓰면서, .md 파일 4개로 역할·목표·제약·검증 기준을 설계했습니다.
>
> **CLAUDE.md** 는 매 세션 자동 로드되는 프로젝트 규칙입니다. 절대 규칙 6개와 작업 순서 8단계가 들어 있고, 예를 들어 "form 태그 사용 금지" 같은 규칙이 코드 곳곳에 반영돼 있습니다.
>
> **PRD** 는 자막 폴백 3단계 전략을 명문화했고, 그게 그대로 코드의 `transcript_ok / metadata_fallback / failed` 상태값이 됐습니다.
>
> **SUMMARY_SCHEMA.json** 은 LLM 출력 JSON 스키마의 진실 공급원입니다. 이 파일을 OpenAI 의 `response_format.json_schema` 에 그대로 전달하기 때문에, 스키마가 진화하면 코드와 LLM 응답이 자동으로 같이 진화합니다.
>
> 마지막으로 **PRESENTATION.md** 에는 이 발표 콘텐츠를 미리 정리해뒀고, 옛 규칙 vs 새 규칙 비교용 두 URL 도 박혀 있어 GitHub 리뷰에서 그대로 노출됩니다.

## 슬라이드 9 — 어려움·해결·한계·개선 (55초)

> 가장 어려웠던 점은 — **"LLM 이 한국어로 요약하면 정해둔 분량이나 구조 규칙을 끝까지 안 지킨다"** 는 거였습니다.
>
> 초기 gpt-4o-mini 결과는 brief 290자에 1문단, 시각 마커 0개로 목표에 한참 못 미쳤습니다. 원인은 프롬프트에 "가이드" 라고 표현한 게 LLM 에게 *권고* 로 받아들여졌다는 점이었습니다.
>
> 그래서 **3겹 가드레일** 을 만들었습니다. 첫째, 프롬프트를 명령조로 강화했고. 둘째, 응답 받은 뒤 validateSummary 함수로 분량·문단·시각 마커를 검증하고 위반 시 자동 재시도하게 했고. 셋째, gpt-5-mini 로 모델을 교체했습니다.
>
> *(슬라이드 우측 표를 가리키며)* 결과적으로 같은 영상에서 brief 가 264자에서 2,122자로, 시각 마커가 0개에서 5개로, 약 8배 정보 밀도가 증가했습니다.
>
> **한계** 도 솔직히 말씀드리면 — RSS 의 최근 15편 제약은 신규 채널 등록 시점에만 진짜 한계입니다. Vercel Hobby 60초 timeout, 자막 음성 인식 오류, 인증 부재 같은 항목들은 모두 v2 에서 풀어갈 계획입니다.
>
> **배운 점** 은 — *"프롬프트 한 줄 잘 쓰는 것보다, 검증·재시도·모델 선택의 가드레일 설계가 신뢰 가능한 LLM 시스템의 핵심"* 이라는 점입니다. 감사합니다.

## 🎤 Q&A 대비 예상 질문 (3분)

| 예상 질문 | 답변 포인트 |
|---|---|
| **"Vercel Hobby 60초 안에 진짜 들어가나요?"** | "네, 실측 50.6초입니다. process limit=2, summarize limit=1 로 보수적으로 잡아둬서 안정적으로 들어갑니다. 다만 backlog 가 쌓이면 따라잡는 데 시간 걸려서, 운영 확장 시 Pro 업그레이드가 자연스럽습니다." |
| **"LLM 비용은 얼마인가요?"** | "영상당 약 $0.025. 월 100편 처리 기준 $2.5, 한국 돈 약 3,300원입니다. Supabase 와 Vercel 은 무료 tier 안에 있어서 실질 운영비는 OpenAI 만 들어갑니다." |
| **"환각 방지는 어떻게?"** | "두 단계입니다. (1) OpenAI structured outputs strict 모드로 JSON Schema 위반 응답 자체를 막고, (2) 시스템 프롬프트에 '자막에 근거 없으면 빈 값으로 둬라' 명령. 실제 곽재식 영상에서 LLM 이 1명만 출연자로 잡고 나머지 2명은 거론된 인물로 분리한 게 그 증거입니다." |
| **"왜 Anthropic Claude 가 아니라 OpenAI?"** | "structured outputs 의 strict 모드가 OpenAI 가 더 안정적이라 채택. Anthropic 도 tool use 로 비슷한 효과 가능하지만 코드 복잡도 차이가 있어서 MVP 단계에선 OpenAI 가 빨랐습니다." |
| **"코너 단위 구독은 정확히 어떻게?"** | "subscriptions 테이블에 channel_id + include_keywords + exclude_keywords 를 두고, RSS 의 영상 제목과 설명문을 합친 문자열에 AND 매치합니다. 예를 들어 이강민의 잡지사 채널에 `#곽재식` 한 단어를 include 로 두면, 곽재식 회차만 자동 인입됩니다." |
| **"매일 cron 만으로 충분한가? RSS 15개 제약은?"** | "정상 운영에서는 충분합니다. 채널이 하루 16편 이상 올리지 않는 한 회전이 일어나기 전에 잡힙니다. 진짜 한계는 *신규 채널 등록 시 그 채널의 과거 영상* 인데, 이건 YouTube Data API 도입으로 해결 가능하지만 쿼터·인증 부담이 있어서 v2 후보입니다." |
| **"왜 Next.js 인가?"** | "API Routes 로 백엔드까지 통합되니까 프로젝트 1개로 풀스택을 다룰 수 있고, Vercel 과 통합 배포가 가장 빠릅니다. 학습 비용 대비 결과가 큽니다." |
| **"발표 자료 끝에 옛/새 규칙 비교 URL 이 박혀 있던데, 어떻게 보존?"** | "8Khu3IoZric (옛) 은 일부러 재요약 하지 않고 그대로 둠. muA-MB0k7SE (새) 는 channelId + force=true 옵션으로 그 1편만 재요약 했습니다. 그래서 한 채널 안에 옛·새 두 결과가 공존합니다." |
