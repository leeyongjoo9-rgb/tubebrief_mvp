## 1. 서비스 개요

유튜브 구독자가 매일 쏟아지는 신규 영상 중 무엇을 볼지 빠르게 판단하지 못하는 문제를, **자동 감지 + 구조화된 LLM 요약** 으로 해결한다. 사용자는 한 번 채널을 등록해두면 매일 0시 cron 이 신규 영상을 가져와 자막을 추출하고 OpenAI 의 structured outputs 으로 JSON Schema 가 강제된 요약을 생성한다. 대시보드에서 헤드라인·본문·출연자·언급된 인물·기업·주제·타임스탬프 점프 링크까지 한 장으로 볼 수 있다.

## 2. 문제와 사용자

| 항목 | 내용 |
|---|---|
| 사용자 | 유튜브 구독 채널이 많고 (8개+) 콘텐츠 중심 시청을 하지만 시간이 부족한 개인 |
| 상황 | 매일 채널마다 1~5편 신규 영상이 올라오는데 다 보기 어렵다. 어떤 영상에 시간을 투자할지 모른다 |
| 기존 한계 | 유튜브 기본 자막은 길고 비구조화. 한 문단 캡션은 정보가 부족. 유튜브 알고리즘 추천은 본인 관심사와 안 맞을 때 있음 |
| 차별점 | 단순 1줄 요약이 아닌 **헤드라인·본문(2000자대)·출연자·거론된 인물·기업·주제·시각 점프 링크** 까지 분리된 한 장짜리 보고서. 본인이 구독한 채널만 깨끗하게 모임 |

## 3. 핵심 기능

| 기능 | 설명 |
|---|---|
| **자동 감지 cron** | Vercel Cron 일 1회. RSS 폴링 → 자막 추출 (실패 시 메타데이터 폴백) → LLM 요약 자동 실행 |
| **코너 단위 구독** | 한 채널 안 여러 코너 중 특정 호스트·게스트만 키워드 AND 필터로 매치. **제목 + RSS 설명문** 함께 검색 (예: 이강민의 잡지사 → `#곽재식` 해시태그가 설명문에 있을 때 그 회차만) |
| **출연자 ↔ 거론 인물 분리** | LLM 이 "직접 출연" 과 "대화 중 거론된 인물" 을 별도 필드로 분리. 환각 방지 규칙 + 채널 호스트 자동 제외 |

## 4. 기술스택

| 영역 | 선택 | 선택 이유 |
|---|---|---|
| Framework | Next.js 16 (App Router) | API Routes 로 백엔드까지 단일 프로젝트. Vercel native |
| 언어 | TypeScript | LLM 응답 스키마와 코드 타입 일치, 런타임 오류 ↓ |
| 스타일 | Tailwind CSS v4 + shadcn/ui | AI 코딩 친화적, 디자인 시스템 즉시 사용 |
| DB | Supabase (PostgreSQL) | `jsonb` 컬럼으로 LLM 출력 그대로 저장. 무료 tier 충분. RLS 로 보안 |
| LLM | OpenAI `gpt-5-mini` | structured outputs (JSON Schema strict 모드) 가 환각 방지에 필수. 한국어 분량/마크업 준수도 검증 통과 |
| 자막 | `youtube-transcript` + RSS 폴백 | 자막 추출 라이브러리. 클라우드 IP 봇 차단 회피용 폴백 설계 |
| 신규 감지 | YouTube RSS | API 키·쿼터 불필요. 무료, 안정적 |
| XML 파서 | `fast-xml-parser` | RSS 의 `media:`/`yt:` 네임스페이스 안전 파싱 |
| 배포 | Vercel (Hobby) | Cron + 함수 + 정적 호스팅 통합. GitHub 자동 빌드 |

## 5. MCP / API 활용

| 사용한 API | 활용 목적 | 연동 흐름 |
|---|---|---|
| **OpenAI Chat Completions (`gpt-5-mini`)** | 자막 → 구조화 JSON 요약 생성 | 사용자 입력 (채널 등록) → cron 트리거 → 자막 추출 → 시스템 프롬프트 + Summary JSON Schema 동봉하여 API 호출 → 응답을 Supabase `videos.summary` (jsonb) 에 저장 → 대시보드 렌더링 |
| **Supabase REST + JS SDK** | 채널·구독·영상·자막·요약 저장과 조회 | 서버 API Route 에서 service role key 로 접근. 클라이언트는 publishable key 만 (RLS 보호) |
| **YouTube RSS Feed** | 채널 신규 영상 감지 | `feeds/videos.xml?channel_id=...` 폴링 → XML 파싱 → 제목+설명문에 구독 키워드 매치 여부 검사 → 신규 만 DB INSERT |
| **YouTube 영상 페이지 스크래핑** | URL → channel_id 추출 (다양한 형식: @handle, /channel/UC, 동영상 URL 모두 처리) | 6개 정규식 fallback chain (`<link rel="canonical">`, `og:url`, Schema.org meta, JSON `"channelId"` 등) |

**한계 / 대응**
- LLM 비용: 영상당 ~$0.025. 검증·재시도 루프 1회로 비용 2배 위험 → `temperature: 0.2`, prompt 강화로 retry 최소화
- Vercel Hobby 60초 timeout: 한 cron 호출에 process(limit=2) + summarize(limit=1) 보수적 처리. backlog 누적 시 채널별 우선 처리 옵션 (`channelId`) 활용

## 7. 개발기록 — 가장 어려웠던 점과 해결

**문제: LLM 이 한국어로 요약하면 약속한 분량/구조를 끝까지 안 지킴**

- **원인 분석**: 초기 `gpt-4o-mini` 가 한국어 출력에 보수적. brief 평균 290자 (목표 1000자), 한 문단으로 묶음 (목표 5~8 문단), `(mm:ss)` 시각 마커 0개 (목표 2~5개). 프롬프트에 "가이드" 라고 명시한 게 LLM 한테 "권고" 로 받아들여진 것
- **해결 과정**:
  1. 프롬프트를 명령조로 강화 — "반드시 N자 이상" + "반드시 \n\n 으로 분리" + "최소 2~5곳 시각 삽입"
  2. 그래도 안 됨 → 검증 함수 `validateSummary` 추가. brief 길이·문단 수·시각 마커 부족 시 *"이전 응답이 규칙을 위반했다, 같은 JSON 스키마로 다시 작성하라"* 메시지를 추가해 자동 재시도 (MAX 2회)
  3. 그래도 분량은 모자람 → `gpt-5-mini` 로 모델 교체. 단번에 brief 1800자+ 8문단+ 시각 마커 다수 출력
- **결과**: 같은 자료 (인터넷 검열 영상 13641자 자막) 에 대해 옛 규칙 264자 / 1문단 / 시각마커 0 → 새 규칙 2,122자 / 10문단 / 시각마커 5개 + 기업 7개 추출. **약 8배 정보 밀도 증가**
- **배운 점**: "프롬프트 → LLM 한 번 호출 → 결과 받기" 는 한 컴포넌트일 뿐. **후처리 검증·재시도 루프 + 모델 선택** 이 함께 가야 신뢰할 수 있는 시스템이 됨. 강의 시간에 배운 *"가드레일 두 겹"* 개념 그대로

## 8. 한계와 개선 방향

| 한계 | 향후 개선 |
|---|---|
| **RSS 의 "최근 15개" 제약** — 채널의 그 이전 영상은 RSS 에서 사라져 못 잡음 | YouTube Data API (`search.list`) 도입. 단, 쿼터·인증 부담 |
| **Vercel Hobby 60초 / 일 1회 cron 제약** — backlog 가 쌓이면 따라잡는 데 수십 일 | Pro 업그레이드 ($20/월) 로 300초 timeout + 무제한 cron, 또는 외부 워커 (Railway, Render) 로 분리 |
| **자막 음성 인식 오류** — "곽재식" 이 "곽제식" 으로 음역되는 케이스 | description 의 `#곽재식` 해시태그를 정답 표기로 LLM 한테 주는 보정 (검토 중) |
| **인증 부재** — 배포 URL 공개. 누구나 채널 추가/삭제 가능 | Vercel Password Protection 또는 Supabase Auth (현재는 MVP 단일 사용자) |
| **검색·필터·텍스트 내보내기** — 영상이 더 쌓이면 필요 | v2 후보 |
| **모바일 최적화** — 데스크탑 우선 디자인 | v0.dev 기반 재설계 (수업에서 배운 도구) |
